This document explains exactly what is published, what is kept private, and how an external auditor can verify the integrity of the published research log.

## What we publish

- [`../model_hashes/shield_v1.json`](../model_hashes/shield_v1.json) — frozen identity of the model.
- `daily_predictions/YYYY/MM/YYYY-MM-DD.json` — one file per trading day with the weights that the model output before market execution that day.
- `monthly_reports/YYYY-MM.md` — narrative monthly summary published at the end of each calendar month.

## What we keep private

- The frozen model checkpoint (`.zip`).
- The VecNormalize statistics file (`.pkl`).
- Intermediate features fed to the model.
- Source code of the risk-management layers and the data pipeline.

The model's behavior is fully constrained by its hash plus the daily prediction stream — those are sufficient to verify that the live signals were generated by the same frozen artifact across the entire history.

## What you can verify

### 1. Model immutability

Compare the SHA256 of the published checkpoint (when made available for verification) against the value in [`../model_hashes/shield_v1.json`](../model_hashes/shield_v1.json). If they match, the model has not changed since the freeze.

### 2. No post-hoc fabrication

For every `daily_predictions/YYYY/MM/YYYY-MM-DD.json` file, inspect the git commit timestamp on GitHub blame. The commit must precede 15:30 ET on the same trading day. If a prediction file was committed after market close, it is not part of the track record.

### 3. Live execution matches the published weights

Once Alpaca account statements are published, cross-reference the `weights` block of each daily prediction with the trades executed on the broker side that same afternoon. They must reconcile within market-impact tolerance.

## Storage policy

The frozen checkpoint and VecNormalize file are stored in S3 with Object Lock in **COMPLIANCE mode** and a **5-year retention period**. COMPLIANCE mode is non-overridable: no user, including the AWS root account, can modify or delete the protected objects until the retention window expires. This guarantees Write-Once-Read-Many (WORM) integrity meeting SEC Rule 17a-4(f), FINRA Rule 4511, and CFTC Regulation 1.31 — the standards demanded by institutional family office due diligence.

GOVERNANCE mode (which would allow administrators with `s3:BypassGovernanceRetention` to override) is explicitly NOT used: it would void the cryptographic trust framework of this repository, since an external auditor could not rule out retroactive modification of the published artifacts.

The 5-year retention covers the v1 parallel-run window (until MinTRL-95% confidence is satisfied after v2 launch, typically 6-24 months) plus a comfortable audit margin for any retrospective due diligence by allocators.

## Statistical reporting methodology

All headline risk and performance metrics (Sharpe ratio, Maximum Drawdown, Calmar ratio, Sortino ratio) reported anywhere in this repository — README, monthly reports, dashboard — are **not reported as point estimates**. Each metric is reported with its **95% confidence interval derived from Stationary Block Bootstrap** (Politis & Romano, 1994) applied to the daily portfolio return series.

### Procedure

- Resamples: 10,000.
- Block length: calibrated automatically from the lag-1 autocorrelation of squared returns of the same series. Calibrating on squared returns rather than raw returns preserves the volatility-clustering structure that the model's path-dependent components (CBF-QP projection, EWMA-20/60 dual vol-targeting, BOCPD changepoint detector) rely on at inference time.
- Reproducibility: every bootstrap output is regenerated by `train/bootstrap_shield_metrics.py` (script lives in the private trading-system repo). The published artifact in this repository is the JSON output `model_hashes/bootstrap_metrics.json`, containing the lower bound, point estimate, and upper bound of each metric.

### Why not IID bootstrap

A vanilla IID (independent-and-identically-distributed) bootstrap shuffles individual return observations. That destroys the serial dependence — autocorrelation and volatility clustering — that Shield's path-dependent risk layers were designed around. Confidence intervals from an IID bootstrap on this kind of series are systematically too narrow and would overstate confidence.

### Why not stochastic policy evaluation

An alternative would be to evaluate the policy under multiple stochastic action samples instead of the deterministic action. We do not do that for two reasons. First, the production deployment uses the deterministic action, so stochastic evaluation introduces epistemic noise that is irrelevant to what is actually executed. Second, repeated stochastic evaluation around the operating point is empirically associated with policy collapse: the variance estimator loses its grounding because the same policy can drift across the evaluation samples. The bootstrap on realized returns sidesteps both issues.

## Live reconciliation

This section describes the post-deployment reconciliation reports that will be published to compare modeled assumptions with empirical execution.

- **Quarterly weights reconciliation (from month 3 of live execution).** Each quarter, the published pre-trade weights for every trading day in the quarter are compared against the realized end-of-day positions reported on Alpaca account statements. Differences attributable to fractional-share rounding, partial fills, or trading halts are itemized. The reconciliation report is published under `monthly_reports/reconcile_QYYYY.md`.
- **Slippage reconciliation (from month 6 of live execution).** The Almgren-Chriss market-impact parameter `eta = 0.1` used in the backtest cost model is compared against `empirical_eta`, the impact coefficient implied by realized fills versus mid-quote at decision time. If `empirical_eta` deviates by more than 50 percent from the modeled value, a technical addendum is published explaining the deviation and any consequences for the validation metrics.
- All reconciliation reports are committed to `monthly_reports/` with the same append-only discipline as the daily predictions.

## Hash verification — manual procedure

When the published model is made available for verification, anyone can run the following on a Unix shell to recompute its hash and compare with the published value:

```
curl -O https://<published-model-url>/TQC_Shield_v1.zip
sha256sum TQC_Shield_v1.zip
# compare with model_hashes/shield_v1.json -> model_sha256
```

If the two hexadecimal strings match exactly, the file is bit-identical to the one declared in this repository at freeze time.

## Regulatory positioning

This research project is structured as an unregulated publisher of quantitative financial research, NOT as an investment advisor. To maintain this regulatory positioning under both EU MiFID II (CNMV interpretation) and US SEC Investment Advisers Act of 1940 (Publisher's Exclusion under Lowe v. SEC), the following operational guardrails apply:

1. **Zero personalization**: signals are identical for every subscriber. No subscriber profiling, no risk tolerance assessment, no investment goal questionnaires (no KYC/MiFID II suitability tests).
2. **Zero individualized communication**: the operator does not respond to subscriber emails seeking personalized guidance ("should I allocate X% to this signal?", "is this right for my situation?"). Such interactions would void the publisher exemption.
3. **Strict regular schedule**: signals are published on a fixed automated schedule (15:30 ET each trading day) to satisfy the SEC's "general and regular circulation" requirement under the Publisher's Exclusion.
4. **Marketing integrity**: this project is positioned exclusively as broad quantitative research, not personalized wealth management. Marketing copy on subscription pages, GitHub README, and all external communications consistently reflects this positioning.

A subscriber retains full responsibility for evaluating whether and how to act on the published signals. The operator does not provide investment advice in any jurisdiction.
